## Goal

Support common MySQL binary prepared-statement flow without handle leakage.

## Work

- [x] Implement prepare, execute, send-long-data, reset, fetch policy, and close.
- [x] Map client statement IDs to upstream IDs.
- [x] Preserve parameter and column metadata.
- [x] Handle null bitmap, signedness, temporal, decimal, blob, and long values.
- [x] Pin sessions with prepared statements for first implementation.
- [x] Release all upstream handles on disconnect or reset.

## Tests

- [x] Invalid handle and duplicate-close tests.

## Done when

- [x] Supported drivers pass prepared statement corpus.

- [x] Evidence recorded and status set to `DONE`.

## Evidence

Commit/PR: next session-correctness checkpoint

Drivers: `go-sql-driver/mysql` v1.9.3 passes the native binary path on MariaDB 11.7.2 and Oracle MySQL 8.4.10. The corpus covers client-local statement IDs, parameter counts, execution column metadata, null bitmap, signed integers, Unicode, DECIMAL, DATE, TIME(6), DATETIME(6), MEDIUMBLOB, a 5 MiB `COM_STMT_SEND_LONG_DATA` value, insert ID, queries, and a prepared update rolled back inside a transaction. Long-data memory is globally bounded to 16 MiB per statement and cleared after execute/reset. Handles close on command or disconnect. Duplicate close is idempotent, unknown handles fail closed, and fetch/cursors return a stable unsupported error. Prepare-time result-column metadata is intentionally zero for the current Go-driver lane; execution metadata is exact. Other client drivers remain unproven and therefore unsupported.
